# Notion 監査ログレビュー スキル

Notion 監査ログ CSV (`Space-*-AuditLog-Export-*.csv`) を対象に、セキュリティ観点で「どこを見れば良いか」を絞り込んだ Markdown サマリーを生成する。CSV 自体は数万〜十数万行になりうるため、決定論的な集計 (フィルタ・件数・バースト検出・国外IP抽出・夜間検出・Markdown 整形) は同梱スクリプトに任せ、LLM は **観点と対象アクティビティ種別の選定**、および **結果に対する所感・確認事項のナレーション** に集中する。

## 全体ワークフロー

監査は 3 ステップで進める。スクリプトを使い回すことで、毎回 14 万行の CSV を Python ロジックを書き直してパースし直す必要をなくす。

1. **棚卸し** — `scripts/list_activity_types.py` でアクティビティ種別の一覧を取得し、ユーザーと一緒にどれを「監査対象 / 除外」にするか決める
2. **解析** — 観点定義を JSON で書き、`scripts/analyze_audit_log.py` を実行 → JSON 集計結果と Markdown サマリーが書き出される
3. **レビュー** — Markdown を読み、必要なら観点を追加/削除して再解析

「初回」なら 1 から、「観点を変えて再生成」なら 2 から始める。

## ステップ 1: アクティビティ種別の棚卸し

```bash
python <SKILL_DIR>/scripts/list_activity_types.py <CSV_PATH>
```

出力は JSON で、全行数・期間・アクティビティ種別ごとの件数・代表アクター・代表説明文がリストされる。これを見ながら、ユーザーに以下を確認する。

- どの観点でレビューしたいか (外部公開 / 削除 / 認証 / 権限変更 / API / AIエージェント / 運用設定 / その他)
- 各観点に **どのアクティビティ種別を含めるか** (種別名はワークスペースのロケールや Notion のアップデートで揺れるので、毎回 LLM が意味から判定する)
- 明示的に除外したい種別があるか (運用上仕方ない操作: 通常編集、コメント、サジェスト、エクスポートなど)

`references/perspectives_template.md` に、よくある観点と対応しそうな種別名の手がかりがあるので参照する。ただしテンプレに完全には依存せず、`list_activity_types.py` の出力に出てくる **実際の種別名** から選ぶこと。Notion 機能追加で未知の種別名が出てきても、意味からカテゴリを推定してよい。

ユーザーに「抽出対象 / 除外」の案を提示して合意を得てから次に進む。

## ステップ 2: 観点定義 JSON を書いて解析実行

合意した観点と対象種別をもとに、設定 JSON を作る。スキーマは `references/config_schema.md` を参照。最小例:

```json
{
  "perspectives": [
    {"name": "外部公開・共有設定", "etypes": ["ページのアクセス許可を更新"]},
    {"name": "削除・アーカイブ操作", "etypes": ["ページをゴミ箱に移動", "ファイルを削除"]},
    {"name": "不正ログイン・認証", "etypes": ["ログイン", "ログアウt"]},
    {"name": "権限変更", "etypes": ["メンバーのロールを変更", "ゲストを削除"]},
    {"name": "API・インテグレーション・AIエージェント", "etypes": ["外部/パブリックインテグレーションを接続"]}
  ],
  "critical_etypes": ["外部/パブリックインテグレーションを接続"],
  "system_actors": ["Notion App", "Notion AI", "Automation", "Notion Calendar"],
  "night_etypes": ["ページのアクセス許可を更新", "メンバーのロールを変更", "外部/パブリックインテグレーションを接続"],
  "burst": {"etype": "ページを閲覧", "per_minute": 100, "per_5min": 300},
  "domestic_country_codes": ["JP"]
}
```

設定の意味:

| キー | 役割 |
|---|---|
| `perspectives[]` | 観点ラベルと、それに含めるアクティビティ種別の完全一致リスト |
| `critical_etypes` | 件数によらず最優先で報告する種別 (例: 外部インテグレーション接続) |
| `system_actors` | この件名は自動処理として件数集計から除外するアクター |
| `night_etypes` | JST 22:00–06:00 で発生したら「夜間リスク操作」として別枠で報告する種別 |
| `burst` | この種別の操作が短時間に集中したらバーストとして検出 (主に大量閲覧) |
| `domestic_country_codes` | これ以外の国コードからの **ログイン** を「国外ログイン」として別枠報告 |

設定を書いたら次のように実行する:

```bash
python <SKILL_DIR>/scripts/analyze_audit_log.py <CSV_PATH> \
  --config <config.json> \
  --out-dir <output_dir>
```

`<output_dir>` (典型的には CSV と同じディレクトリ) に以下が生成される:

- `audit-summary.md` — 人が読むためのレポート (観点別件数、要確認イベント、バースト、国外ログイン、夜間操作、除外種別一覧)
- `audit-summary.json` — 同じ内容の機械可読版 (LLM が次のイテレーションで再利用したり、ピンポイントで深掘りするのに使う)

## ステップ 3: レビューと反復

生成された `audit-summary.md` を読み、ユーザーに提示する。**スクリプトはあくまで集計まで**。LLM はそこから:

- 「最優先で確認すべき重大イベント」が本当に critical か (実行者・接続先・時間帯から判断)
- 観点ごとに件数が多すぎて読めない → 観点を分割するか、特定の種別を除外するかをユーザーに提案
- 「除外したアクティビティ種別」一覧を見て、見落としがないか (新出の種別を観点に追加すべきでないか) を再確認

を判断する。観点を変えたい場合は config JSON を編集して **ステップ 2 から** やり直すだけ。CSV パースをやり直す必要はない (とはいえスクリプト全体の所要は数秒〜十数秒で十分速い)。

## 出力する Markdown レポートの構成

スクリプト `analyze_audit_log.py` が常に以下の章立てで生成する。順序は固定 (ユーザーが「§N を §M の前に」と言わない限り変えない)。

```
# Notion 監査ログ サマリーレポート
## 0. 一行所感         ← critical / 権限 / エージェント / 削除 / 夜間 / 国外 を 1 段落で要約
## 1. 観点別 件数サマリ ← 表で各観点の件数 + 内訳
## 2. 管理者が確認すべき事項 ← 各カテゴリで具体的アクションをリスト
## 3. 観点別 要確認イベント  ← 各観点ごとに表 (>30件はユーザ×種別サマリ + <details>で上位30件)
## 4. 大量操作 / バースト
## 5. 国外IPからのログイン
## 6. 夜間帯 (JST 22:00–06:00) のリスク操作
## 7. 除外したアクティビティ種別  ← 全種別件数 (対象外のもの) を表で提示
```

ユーザーから「順序を変えて」「この章は不要」と指示があれば、`analyze_audit_log.py` の出力をベースに編集する (Markdown を直接編集すれば十分; スクリプトを書き換える必要はない)。

## 設計上の要点

- **アクター集計から除外するシステムアカウント** — `Notion App` / `Notion AI` / `Automation` / `Notion Calendar` は自動処理。`ページを完全に削除` `ページをゴミ箱から削除` `アクセス不可` などは大半がこれらによる。`system_actors` で除外することでバースト誤検知を防ぐ。
- **アクティビティ種別の完全一致で判定** — 「ページ名に "セキュリティ設定" が含まれる」のような部分一致は誤検知の温床。判定は必ず `etype` 完全一致で行う (スクリプトもそう実装してある)。
- **MCP 判定** — アクティビティ名ではなく `MCPリダイレクトURL` 列が `N/A` 以外かで判定する (今回はサマリーには含めていないが、深掘りする場合は JSON 側で見られる)。
- **JST 変換** — CSV の `日付と時刻（UTC）` を JST に変換して扱う。夜間判定・期間表示すべて JST。
- **海外アクセス全般 ≠ 国外ログイン** — `ページを閲覧` などは Notion サーバ経由で海外 IP に見えることがある。`国外IP` として別枠報告するのは **ログイン (`ログイン` 種別)** に限定する。
